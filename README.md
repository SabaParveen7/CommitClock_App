# CommitClock

GitHub commit alarm + streak companion app, built per `CommitClock_PDR_v2.docx`.

This zip contains a **complete, runnable Kivy source project** implementing
every feature in the PDR: schedule storage, the alarm engine, the full-screen
alarm UI with 5 character moods, the schedule/heatmap screen, the streak
system, motivational messages, and alarm-tone settings.

---

## 1. Run it right now (desktop)

```bash
pip install -r requirements.txt
python main.py
```

This opens the app in a window. Since the bundled schedule (`data/commits.json`)
is generated for real dates in June–July 2026, the alarm won't fire immediately
on a fresh run unless your system clock happens to match a scheduled entry.
To **test the alarm flow right now**, do either of:

- Temporarily edit one entry's `"date"`/`"time"` in `data/commits.json` to a
  minute or two from now, then run the app and wait — or
- In `alarm_service.py`, call `self.check_now(now=<a datetime you choose>)`
  from a quick test script to fire the alarm immediately:

  ```python
  from datetime import datetime
  from alarm_service import AlarmService
  svc = AlarmService(on_alarm=lambda e: print("ALARM:", e))
  svc.check_now(now=datetime(2026, 6, 12, 9, 0))   # matches the first entry
  ```

## 2. Project structure

```
CommitClock/
  main.py                    — App entry point, screen wiring
  alarm_service.py           — Time-checking engine (foreground-only, see limitations)
  mood_engine.py             — Mood -> sprite + message resolution
  streak_tracker.py          — Streak, heatmap, and schedule read/write logic
  screens/
    schedule_screen.py       — Home screen: table, progress bar, heatmap, idle character
    alarm_screen.py          — Full-screen alarm: character, Submit, Snooze
    settings_screen.py       — Alarm tone picker
  widget/
    commit_widget.py         — Stub for the home-screen widget (stretch goal, not built)
    android_service_notes.md — What's needed to finish the widget + real background service
  data/
    commits.json             — 160 pre-generated commit entries (June 12 – July 11, 2026)
    messages.json            — Motivational message pools per mood
    settings.json            — Selected alarm tone + streak record (read/write at runtime)
  assets/
    character_*.png          — 5 placeholder mood sprites (generated, see below)
    *.wav                    — 4 placeholder sounds: 3 alarm tones + 1 success chime (synthesized)
  tools_generate_schedule.py — Regenerate commits.json
  tools_generate_sprites.py  — Regenerate placeholder character art
  tools_generate_sounds.py   — Regenerate placeholder sounds
  buildozer.spec             — Android build config
  requirements.txt
```

## 3. Customize before your real campaign

**Replace the placeholder commit messages.** `data/commits.json` currently has
generic text like `"Commit #1: Refactor UI layer"`. Open
`tools_generate_schedule.py`, replace the `PLACEHOLDER_VERBS` / `PLACEHOLDER_AREAS`
logic (or just hand-edit the JSON directly) with your actual 160 MedSim commit
messages, then re-run:
```bash
python tools_generate_schedule.py
```

**Replace the placeholder art and sound.** Everything in `assets/` was
generated by `tools_generate_sprites.py` (simple geometric chibi shapes) and
`tools_generate_sounds.py` (synthesized tones) so the app is fully runnable
out of the box without needing external files. The PDR calls for proper
hand-drawn/animated sprite sheets and real recorded tones — swap the files in
`assets/` (keep the same filenames, or update `mood_engine.py` /
`screens/settings_screen.py` if you rename them).

**Adjust message pools.** Edit `data/messages.json` directly — no code
changes needed, the app reads it at runtime.

## 4. Building the Android APK

This sandbox cannot build or test an actual `.apk` — Buildozer needs the
Android SDK/NDK and a full Linux build toolchain that isn't available here.
`buildozer.spec` is fully configured and ready to go on your own machine:

```bash
pip install buildozer cython
sudo apt install -y git zip unzip openjdk-17-jdk python3-pip autoconf libtool pkg-config
buildozer -v android debug
```

The first build downloads the SDK/NDK and will take a while. The resulting
`.apk` will be in `bin/`. Install it with:
```bash
adb install bin/commitclock-2.0-debug.apk
```

## 5. Known limitations — read before relying on this for the real campaign

1. **Background alarm reliability.** `alarm_service.py` uses Kivy's `Clock`,
   which only runs while the app process is alive. This is correct and
   sufficient for the desktop demo and short backgrounding, but **a real
   30-day campaign needs a true Android foreground Service** (declared via
   `services =` in `buildozer.spec`, with its own Java/Pyjnius entry point,
   a persistent notification, and the WAKE_LOCK/FOREGROUND_SERVICE
   permissions already listed in `buildozer.spec`). That native glue isn't
   included — see `widget/android_service_notes.md`. This is flagged as the
   #1 risk in the PDR's risk table for good reason; test it on a real device
   early, well before day 1 of the campaign.
2. **Home-screen widget is not built** (PDR 5.8 stretch goal). Stubbed in
   `widget/commit_widget.py` with notes on what's required.
3. **Placeholder art and sound.** Functional but simple — meant to make the
   app runnable immediately, not as final visual/audio design.
4. **Lock-screen full-screen alarm permission** (`SYSTEM_ALERT_WINDOW`) is
   declared in `buildozer.spec` but Android 10+ also needs a full-screen
   intent notification for reliably waking the screen from a background
   service — part of the same native service work in point 1.

## 6. Data schema reference

`data/commits.json` — one entry per scheduled commit:
```json
{"id": 1, "date": "2026-06-12", "time": "09:00", "message": "...", "status": "pending"}
```
`status` is one of `pending`, `done`, `missed` — updated automatically by
`streak_tracker.py`.

`data/settings.json`:
```json
{"alarm_tone": "classic_alarm.wav", "current_streak": 0, "longest_streak": 0, "milestones_seen": []}
```
